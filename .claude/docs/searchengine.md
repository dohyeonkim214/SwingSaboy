markdown_content = """# 유튜브 Data API v3 기반 스윙댄스 영상 자동 수집 엔진 설계서

본 문서는 백엔드 서버를 활용하여 유튜브 Data API v3의 할당량 제한을 우회하고, 정적 페이지 구조에서 안전하고 확장성 있게 스윙댄스 영상을 자동 수집·제공하기 위한 검색 엔진 아키텍처 설계서입니다.

---

## 1. 개요 (Overview)
* **목적:** 스윙댄스 앱 내 카테고리별(린디합, 지터벅, 발보아, 블루스, 찰스턴 등) 최신 및 추천 영상을 자동 수집하여 유저에게 제공.
* **배경:** 프론트엔드(정적 페이지)에서 유튜브 API를 직접 호출할 경우 API 키 노출 위험 및 일일 할당량(10,000 유닛)의 급격한 소모 문제가 발생함.
* **해결책:** 백엔드 서버에 독립적인 스케줄러(Batch Scheduler)를 도입하여 주기적으로 데이터를 데이터베이스(SQL)에 캐싱하고, 프론트엔드에는 자체 가공된 API를 제공하는 구조로 전환.

---

## 2. 유튜브 Data API v3 분석 및 제약 사항
유튜브 API는 사용량에 따른 과금 방식이 아닌, 일일 제공되는 **무료 할당량(Quota)** 제한 방식으로 운영됩니다.

### 2.1 할당량 구조
* **기본 일일 할당량:** 10,000 유닛 (매일 PDT 자정 초기화)
* **주요 메서드별 할당량 비용:**
  * `search.list` (키워드 검색, 채널 내 검색 등): **100 유닛** (매우 높음)
  * `videos.list` (영상 상세 정보, 조회수, 카테고리 등): **1 유닛**
  * `playlists.list` / `playlistItems.list` (재생목록 조회): **1 유닛**

### 2.2 실시간 클라이언트 호출의 한계
유저가 앱에서 검색창을 통해 직접 유튜브 API를 호출하는 구조(`search.list`)일 경우:
$$\text{최대 검색 가능 횟수} = \frac{10,000 \text{ 유닛}}{100 \text{ 유닛/검색}} = 100\text{회}$$
앱 전체 유저를 통틀어 하루에 딱 100번의 검색만 일어나도 서비스가 전면 마비되므로, **백엔드를 통한 데이터 캐싱 및 프록시 처리가 필수적**입니다.
